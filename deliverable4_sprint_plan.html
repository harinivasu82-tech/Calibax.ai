01 Sprint plan — phase overview + task detail

Five phases. Six gate checkpoints — each with a binary pass/fail and a pre-defined escalation action. Three roles running in parallel from Day 3. Note on gate count: Phase 1 · Foundation carries two gates (G1 at End Day 1, G2 at End Day 2) — both are sub-gates within the same phase because Foundation has two distinct critical handoffs: sprint start and schema lock. Every other phase has one gate. The plan is designed to front-load risk: the hardest dependency (data consistency between structured and unstructured corpus) is resolved by Day 4, before the engineer has wired anything to it.

Phase 1 · Foundation
Days 1–2 G1 · End D1 G2 · End D2 (sub-gate)
PM
5 demo questions locked and delivered to DS before 9am Day 1. Questions are derived directly from James Whitfield's words in the SDR memo. They drive the entire data design — data cannot start until these exist.
Deliverable: written question list + data authoring template in squad channel before first standup
DS
Data authoring begins Day 1. Sales & margin fixture (500–800 rows), supply chain incident log, Teams/email corpus. Internally consistent — the Auckland DC delay in the WMS log must match the NZ margin dip in the sales fixture.
Risk: this is the sprint's highest-risk task. It must start before anything else.

All
API schema agreed and locked by end of Day 2. PM chairs, TL present. Schema defines the SSE event structure (token, done, citation object). No ENG UI work continues until schema is signed off. Written to squad channel.
G1
Pass: 5 questions locked, template delivered, LLM API keys live, DS-2 named as contingency. Fail: PM escalates to Calibrax leadership same day. Sprint pauses until resolved.
G2

Phase 2 · Build
Days 3–6 · parallel tracks G3 · End D4
DS
RAG pipeline complete by end of Day 6. Corpus chunked at 200–350 tokens with 40-token overlap, embedded with text-embedding-3-small, stored in ChromaDB locally. k=4 retrieval. All unstructured documents ingested.
Checkpoint: ChromaDB ingestion must be started by end of Day 4 (G3).
DS
Query classifier + system prompt + numeric guardrail. Keyword-based routing (~30 lines). Structured queries → lookup table. Unstructured → RAG. Ambiguous → both. System prompt receives lookup data + RAG chunks + query, returns blended prose with citations. Guardrail post-processor verifies every number in the response exists verbatim in the source context.
ENG
SSE streaming renderer. Consumes token events from FastAPI, appends token-by-token to message in real time. No visible chunking. Builds against stub until DS endpoint is live Day 7.
No model switching after Day 3 — system prompt tuning is model-specific and costs 1–2 days to re-tune.
ENG
Question chips, source attribution panel, Demo Mode, graceful failure UI. 5 chips above input — click populates field, CEO presses Enter. Collapsible citation panel renders from done event. Demo Mode (?demo=true param) serves pre-cached responses for 5 scripted questions with simulated streaming. Fallback UI: calm message, zero stack traces, zero raw error strings.
PM
Pitch deck and commercial ask. Problem framing, solution vision, prototype walkthrough narrative, paid discovery close. Runs in parallel to technical build — not a squad task.
G3
Pass: Lookup JSON complete (40+ query variants). ChromaDB ingestion started. Fail: Second DS activated for Days 5–8. No grace period. PM notifies leadership immediately.
Phase 3 · Integration
Days 7–9 · D8–9 weekend G4 · End D7
DS
FastAPI SSE endpoint live by end of Day 7. POST /api/query streaming token events then done event with citation JSON. Unit tests covering guardrail pass and fail cases. All DS-side components wired together and tested in isolation before handing to ENG.
ENG
Full end-to-end integration. React front-end wired to live DS endpoint. All 5 scripted questions smoke tested end-to-end. Latency measured on demo hardware — first token must appear within 2 seconds.
D8–D9 are weekend days. DS integration QA and ENG smoke testing continue. PM reachable for scope decisions.
G4
Pass: ≥3 of 5 scripted questions returning passing answers end-to-end by end of Day 7. Fail (<3 pass): all nice-to-haves cancelled. Fail (<2 pass): leadership escalation, scope review same day.
Phase 4 · Validation
Days 10–12 G5 · End D12 · pitch go/no-go
PM
15-query red-team on Days 11–12. PM runs a predetermined adversarial query set. Covers: edge cases, unusual phrasings, multi-part questions, off-topic queries, and numeric stress tests. Pass/fail logged per query. All failures briefed to DS + ENG immediately.
Target: ≥10 of 15 off-script queries return plausible, non-embarrassing responses.
DS
Red-team fixes and guardrail stress testing. All numeric failures from red-team patched. Guardrail tested to confirm zero hallucinated numbers escape to UI under adversarial conditions.
ENG
UI polish, visual QA, and Demo Mode final cache. Feedback from red-team observations incorporated. Demo Mode pre-cached responses updated to match any data changes made during validation. Streaming cadence confirmed visually identical between live and cached.
G5
Pass: 5/5 scripted questions pass. Zero visible errors. Zero hallucinated numbers. ≥10/15 off-script plausible. Fail: Day 13 used for emergency fixes. PM makes pitch go/no-go call. If no-go: PM contacts FreshChoice to reschedule.
Phase 5 · Lock & prep
Days 13–14 G6 · End D13 · code freeze

Code freeze end of Day 13. No changes after this point without joint TL + PM written sign-off. Demo environment locked. ENG confirms app runs cleanly on pitch venue hardware — no surprises on the day.
PM
Full dress rehearsal on pitch laptop Day 13. PM walks through the complete demo script while ENG operates the machine. All 5 scripted questions triggered in demo order. Timing measured. Narrative transitions practised.
PM
Day 14: final prep and travel. Final review of pitch materials. Confirm ENG is physically on-site at venue for Day 15. Demo Mode tested on venue WiFi — confirm offline fallback if needed.
G6
Pass: Code frozen. Demo tested on pitch laptop. ENG confirms venue hardware clean. Any post-freeze change: TL + PM written sign-off required. Risk flagged immediately.

PM chairs, TL present
Gate criteria checked — binary pass/fail, not a discussion
Escalation action fires immediately on failure — not deferred
Written record posted to squad channel within 1 hour
Async rules · always on
All tasks tracked on shared board (Notion / Linear)

04 Escalation matrix — pass criteria & failure actions

Every gate has a binary pass condition and a pre-defined escalation action. The PM makes the call on the day — there is no "let's revisit tomorrow." Ambiguity at a gate checkpoint is itself a failure state.
