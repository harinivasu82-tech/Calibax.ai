01 Context — read D1 first

We are building a 14-day pitch prototype for FreshChoice Group — a $2.4B grocery retailer whose CEO cannot answer basic business questions without a 1–2 day human relay across five disconnected data systems. Full problem analysis, user landscape, and market scan are in Deliverable 1.

This document covers what to build, how it should behave, and what the squad needs to know to ship it. The prototype must make James Whitfield say "this is exactly what I've been asking for" in under 20 minutes. Everything below is in service of that moment.

02 Prototype scope

This is a 14-day, pitch-quality prototype — not a production system. Every scope decision is optimised for demo reliability and wow factor, not engineering completeness. Every cut below was made deliberately.

Must have required — no exceptions
Conversational chat interface — CEO types a free-text question and receives a prose answer. Clean, professional web UI. No login screen for the demo.
Pre-computed structured data lookup — ~40–50 query variants covering all 5 demo questions and natural rephrasings. Deterministic, millisecond responses.
RAG pipeline over unstructured corpus — Teams message threads and email summaries chunked, embedded, and stored in ChromaDB. Top-k retrieval on every relevant query.
Query intent classifier — keyword-based routing: numbers/metrics → lookup, people/escalation → RAG, blended → both. Fallback to "both" on any unrecognised query.
LLM synthesis layer — receives lookup data + RAG chunks + query. Returns blended prose with citations. System prompt enforces source-only responding for numeric values.
Numeric hallucination guardrail — post-processing extracts all numbers from LLM response and verifies each exists verbatim in source context. If any fail: graceful fallback triggered.
SSE streaming response — answer streams token-by-token from FastAPI to React. First token must appear within 2 seconds of query submission.
Source attribution panel — collapsible panel beneath each answer showing 1–3 citation references (source name, type, short excerpt).
Graceful failure handler — when guardrail triggers: confident, non-technical message. Zero Python errors visible. Zero hallucinated numbers ever rendered.
Suggested question chips (5) — pre-validated demo questions as clickable chips. Each has a confirmed good answer in the test suite.
Demo Mode toggle — hidden keyboard shortcut or URL param that serves pre-cached responses for the 5 scripted questions with client-side streaming simulation. Off-script queries always use the live system.
Nice to have only if all 5 scripted questions pass by Day 7
Proactive alert feed — 2–3 pre-generated insight cards above the chat input. Powerful for showing push intelligence. Do not build until core is solid.
Role switcher — lightweight toggle between CEO view and Regional Manager view. Excellent for the pitch narrative. One afternoon if core is done.
Follow-up question context — retain the previous answer in LLM context for clarifying follow-ups. Adds depth to the conversational feel.
Out of scope explicitly excluded — do not build
Live API connections to SAP, Salesforce, or any real client system
User authentication, access control, or session management
Mobile interface or responsive layout below 1024px
Full dynamic NL-to-SQL engine — this is Phase 2 in the paid engagement
WhatsApp or Teams bot interface
Data persistence, user history, or conversation storage
Production error handling, logging, or monitoring
03 Key user stories

Primary persona: James Whitfield, CEO, FreshChoice Group. Finance and operations background. Highly commercial. Sceptical of conceptual pitches. Will type his own questions. Must be convinced in 20 minutes.

US-01 Ask a structured data question must have
As the CEO, I want to type "How did fresh produce margins perform in New Zealand last month versus the prior quarter?" and receive an answer with actual percentage figures and a short explanation of what drove the change — without waiting for my finance team.
Acceptance criteria
Response includes at least two numeric values and a trend direction
A 1–2 sentence explanation of the driver is included alongside the numbers
First token appears within 2 seconds. Full response within 10 seconds.
Source attribution panel cites the structured data fixture by name
US-02 Ask about what teams are saying must have
As the CEO, I want to type "What are the top issues my regional managers are escalating this week?" and receive a synthesised summary drawn from Teams and email — not a raw message list, but an intelligent distillation of themes.
Acceptance criteria
Response summarises 2–3 distinct themes from the unstructured corpus
At least one citation references a specific source with date
Response reads as synthesised analysis, not a message dump
No numeric values appear unless present verbatim in the corpus
US-03 Ask a blended cross-source question must have — the wow moment
As the CEO, I want to ask "What's happening with supply chain disruptions and how is it affecting our margins?" and get a single answer connecting financial impact from our data with operational context from what my team has been saying.
Acceptance criteria
Answer explicitly references both a number (from lookup) and a qualitative source (from RAG) in the same response
The two are causally connected in the narrative — not listed separately
Passing example: "The 3.2% margin drop in NZ correlates with the Auckland DC delay flagged by Sarah Chen on 11 March"
Source panel shows citations from both structured and unstructured types
All numbers verified by the guardrail post-processor
US-04 See where the answer came from must have — trust signal
As a CEO who has been burned by inaccurate dashboards before, I want to see exactly which data file or message thread each part of the answer was drawn from — so I can trust the numbers without verifying them myself.
Acceptance criteria
A collapsible "Sources" panel appears beneath every answer
Each source shows: name, type (structured / Teams / email), and a short excerpt
Panel is collapsed by default, expanded on click — does not clutter the main answer
1–3 sources per answer. Never zero on a successful response.
US-05 Use suggested questions must have — demo safety net
As a demo participant, I want to see pre-loaded question chips I can click to trigger reliable, impressive answers — so the demo flows naturally even without scripting every move.
Acceptance criteria
5 question chips visible above the input field on load
Clicking a chip populates the input — CEO presses Enter, keeping him in control
All 5 chips have pre-validated passing answers confirmed during red-team
In Demo Mode, these 5 chips return pre-cached answers instantly
Non-functional requirements
Requirement Threshold Test method
Streaming latency First token ≤ 2 seconds after query submission Stopwatch test on demo hardware, 5 consecutive runs on each of the 5 scripted questions. All must pass.
Hallucination guard Zero hallucinated numbers in any response, ever Guardrail post-processor verified unit-tested. Red-team 30 queries manually checked for numeric accuracy before pitch.
Error visibility Zero Python tracebacks or API error messages visible to user Force guardrail failure deliberately in staging. Confirm only the calm fallback message renders — no stack traces, no raw error strings.
Off-script robustness ≥ 10 of 15 off-script queries return plausible, non-embarrassing responses PM-run red-team session on Day 11–12 using a predetermined 15-query adversarial set. Pass/fail logged per query.
Demo Mode fidelity Cached responses stream at identical visual cadence to live responses Side-by-side comparison on demo laptop. No perceptible difference in token-by-token appearance speed between live and cached.
04 Data requirements

All data is synthetic and fictional. No real FreshChoice data required. Data must be internally consistent — if NZ margins drop in week 3 of the structured fixture, the Teams messages must reference that same event. Data authoring is the highest-risk task in the sprint. It must start Day 1 and complete by end of Day 4.

Dataset Owner Specification
Structured fixture 1
Sales & margin data
DS
500–800 rows. Weekly sales volume and gross margin % by: product category, market (AU / NZ / SG), store format. 14–16 weeks of history. Must include a visible margin dip in NZ fresh produce weeks 10–12.
Format: week_ending, market, category, format, revenue_aud, gross_margin_pct, units_sold
Structured fixture 2
Supply chain log
DS
20–40 rows. Incident records covering distribution disruptions. Must include an Auckland DC event in week 11 that causally links to the NZ margin dip in Fixture 1.
Format: incident_date, dc_name, categories_affected, margin_impact_pct, status, summary_note
Unstructured corpus 1
Teams message threads
DS
4–6 threads, 8–15 messages each. Covering: NZ fresh produce supply issues, cross-market performance, e-commerce growth, staff rostering escalation. Named characters: Sarah Chen (CSO), regional managers for Auckland, Melbourne, and Singapore.
Format: thread_name, date, sender_name, sender_role, message_body
Unstructured corpus 2
Email summaries
DS
6–10 emails. From regional managers to Sarah Chen or James Whitfield covering: weekly ops summaries, supplier delay notifications, margin escalations, one e-commerce update. Must cross-reference the Auckland DC event from Fixture 2.
Format: email_date, sender_name, sender_role, recipient, subject, body
Pre-computed lookup JSON
DS
40–50 query variants mapping keyword sets to pre-computed answer payloads. Enables deterministic, millisecond responses to structured data questions. Must cover all 5 demo questions plus at least 6 plausible rephrasings of each.
Format: query_keywords[], market?, category?, metric_type → answer_payload{value, trend, context, source_ref}
Embedding index
DS
All unstructured documents chunked at 200–350 tokens with 40-token overlap. Embedded with text-embedding-3-small. Stored in ChromaDB (local). Each chunk metadata: source_name, source_type, date, sender_name. k=4 retrieval.
Target: <500ms retrieval latency on demo hardware
05 API contract — agreed Day 2

Schema agreed between DS and ENG by end of Day 2. No changes after Day 6 without PM sign-off. Engineer builds all UI components against this schema from Day 3 onward.

POST /api/query — request
{ "query": string, // raw user input "demo_mode": boolean, // if true, return pre-cached response "session_id": string | null // reserved for future context — pass null now }
SSE stream — response events
// Event 1..N: token stream event: token data: { "token": string } // Final event: metadata event: done data: { "query_type": "structured" | "unstructured" | "blended" | "fallback", "citations": Citation[], "guardrail_pass": boolean // false = graceful failure was served } // Citation object Citation { "source_name": string, // e.g. "NZ Sales Summary Mar 2025" "source_type": "structured_data" | "teams" | "email", "excerpt": string, // max 120 chars "date": string | null }
06 Technical architecture
Data scientist stack DS owns
Data authoring — synthetic fixtures from PM template. CSV → pandas → pre-computed lookup JSON. Days 1–4.
RAG pipeline — chunk + embed Teams/email corpus. ChromaDB local vector store. text-embedding-3-small. k=4 retrieval. Days 4–6.
Query classifier — keyword-based routing function (~30 lines Python). Structured → lookup. Unstructured → RAG. Unknown → both. Day 5.
System prompt + LLM call — single prompt receiving lookup data + RAG chunks + query. Returns prose + citation JSON. GPT-4o or Claude Sonnet via API. Day 6.
Numeric guardrail post-processor — regex extracts all numbers from LLM response. Verifies each exists in source context. If any fail: return graceful fallback payload. Day 6, 0.5 days.
FastAPI SSE endpoint — POST /api/query. Streams token events then sends done event with citation JSON. Day 7.
Engineer stack ENG owns
React / Next.js web app — chat interface with message history, text input, send button. Clean, professional. Days 1–3.
SSE stream renderer — consume token events from FastAPI, append to message in real time. No visible chunking. Days 2–4.
Suggested question chips — 5 chips above input. Click → populate input field. CEO presses Enter. Days 3–4.
Source attribution panel — collapsible panel rendering citation array from done event. Name, type badge, excerpt. Days 4–5.
Graceful failure UI — detects guardrail_pass: false in done event. Renders fallback message. No error stack traces ever. Day 5.
Demo Mode — URL param ?demo=true activates. Serves pre-cached payloads for 5 scripted questions with simulated streaming. Day 5, half day.
Full integration + smoke test — wire all components to live DS endpoint. Smoke test all 5 scripted questions end-to-end. Days 7–9.

08 Assumptions & key decisions

Assumptions are unknowns we are treating as true. Key decisions are choices already made — they are locked and not open for re-discussion during the sprint.

Assumptions
A-01
Simulated data is accepted by the client
James Whitfield understands this is a pitch prototype. The PM will frame this explicitly: "We've built this on a representative data model — connecting to your live SAP and Teams is what the paid engagement delivers."
Why: Live integration would consume 8–10 engineer days on connectors alone. The CEO cannot distinguish well-designed synthetic data from real data in a 20-minute demo.
A-02
Pre-computed lookup is architecturally honest for the prototype
The lookup table replaces dynamic NL-to-SQL for the prototype only. Full NL-to-SQL is the first deliverable in the paid discovery phase. PM owns the narrative bridge: "What you see here uses a structured retrieval layer — in the engagement we'd connect this directly to your SAP queries."
Why: Saves 3 DS days. CEO output is identical. Deterministic and debuggable. A live NL-to-SQL engine under 14-day pressure is a reliability risk we cannot afford.
A-03
5 demo questions locked by 9am Day 1
The PM delivers 5 scripted demo questions and the data authoring template to the DS before 9am on Day 1. Every hour of delay is an hour of risk at the end of the sprint. Questions are derived from Whitfield's exact words in the SDR briefing memo.
Why: The entire data fixture, lookup table, and system prompt are designed around making these 5 questions produce impressive answers. They must drive the data design, not follow it.
A-04
Second DS is a named, available contingency
A second data scientist is identified by name before end of Day 1. Available to start Day 5 if the Day 4 checkpoint shows the RAG pipeline has not started. The contingency must be real and named, not theoretical.
Why: The DS workstream has no buffer across 14 days. This is insurance, not a plan B — the primary DS is expected to deliver.
A-05
Demo Mode is not deceptive
Pre-caching the 5 scripted answers is standard practice for enterprise software demos. The system genuinely does what it shows — Demo Mode removes API latency risk for hero moments only. Off-script questions always use the live system.
Why: Every major enterprise software demo pre-caches its hero scenarios. The alternative — risking a 15-second blank screen or API timeout in the room — is not acceptable.
Key decisions — locked, not open for discussion
D-01
LLM via API — no fine-tuning, no self-hosting
GPT-4o or Claude Sonnet via their respective APIs. Decision locked Day 1, API keys provisioned before standup. No model switching after Day 3 — system prompt tuning is model-specific and re-tuning costs 1–2 days.
Rationale: Fine-tuning takes weeks. API access takes hours. Demo-quality output from a well-engineered system prompt is indistinguishable from a fine-tuned model to a non-technical audience.
D-02
Demo runs locally on a single engineer laptop
Prototype runs locally, not in cloud infrastructure. Internet required for LLM API calls. Demo Mode pre-caches all 5 scripted answers as WiFi fallback. Engineer is physically on-site during the pitch, running the machine.
Rationale: Removes the need for production hardening, deployment pipelines, load testing, and cloud security review — none achievable in 14 days.
09 Sprint gates & checkpoints

These are go/no-go decision points. If a gate fails, the escalation action is immediate — not deferred to the next standup.

End of Day 1
Foundation gate
5 demo questions locked. Data template handed to DS. LLM provider decided, API keys working. Second DS name confirmed. Failure: PM escalates to Calibrax leadership same day.
End of Day 2
API contract gate
Response schema signed off by DS + ENG. TL chairs the session. Schema document shared to squad channel. No schema changes after Day 6 without PM approval.
End of Day 4
DS pipeline checkpoint
Lookup JSON complete. ChromaDB ingestion started. If either condition is not met: second DS activated immediately for Days 5–8. No grace period.
End of Day 7
Integration gate
At least 3 of 5 scripted questions returning passing answers end-to-end. If fewer than 3 pass: scope review, nice-to-haves cancelled. If fewer than 2 pass: escalate to leadership.
End of Day 12
Red-team gate — pitch go/no-go
All 5 scripted questions pass. Zero visible errors. Zero hallucinated numbers. At least 10 of 15 off-script queries return plausible responses. Failure: Day 13 used for emergency fixes. PM decides whether to pitch.
End of Day 13
Code freeze
No code changes after Day 13 without joint TL + PM approval. Demo environment locked. Demo Mode tested on pitch laptop. Engineer confirms environment runs cleanly on venue hardware.
