01 Problem statement

James Whitfield, CEO of FreshChoice Group — a $2.4B grocery retailer with 280+ stores across Australia, NZ, and Singapore — cannot get timely answers to basic business questions. A question like "How did fresh produce margins perform in NZ last month vs the prior quarter?" requires a 1–2 day human relay: CEO → CFO → finance team → SAP extract → Excel analysis → email back.

His data lives in five disconnected systems: SAP (finance), Salesforce (CRM), a proprietary POS (transactions), a legacy WMS (supply chain), and Teams/WhatsApp/email (qualitative operational context). Nobody has a unified view. A previous Tableau rollout failed 18 months ago due to poor data integration and low adoption. The CEO is sceptical of vendor pitches that are "all slides, no substance."

This prototype must make that problem viscerally obvious — and then instantly dissolve it. The CEO's exact words: "I want to ask a question and get an answer — like I'm talking to my smartest analyst who has access to everything." That is what we are building. Not a dashboard. Not a report. A conversation that returns numbers and context, together, in seconds.

This is not a FreshChoice-specific problem. The same pattern — data silos, human relay bottlenecks, slow decisions — repeats across large retailers, FMCG companies, logistics operators, and healthcare networks. The prototype is the proof-of-concept for a reusable enterprise platform.

02 Prototype scope

This is a 14-day, pitch-quality prototype — not a production system. Scope decisions are optimised for demo reliability and wow factor, not engineering completeness. Every cut below was made deliberately to protect quality on what matters in the room.

Must have required — no exceptions
Conversational chat interface — CEO types a free-text question and receives a prose answer. Clean, professional web UI. No login screen for the demo.
Pre-computed structured data lookup — ~40–50 query variants covering all 5 demo questions and natural rephrasings. Returned in milliseconds. Deterministic and debuggable.
RAG pipeline over unstructured corpus — Teams message threads and email summaries chunked, embedded, and stored in ChromaDB. Top-k retrieval on every relevant query.
Query intent classifier — keyword-based routing: numbers/metrics → lookup, people/teams/escalation → RAG, ambiguous or blended → both. Fallback to "both" on any unrecognised query.
LLM synthesis layer — receives lookup data + RAG chunks + user query. Returns a blended prose answer with citations. System prompt enforces source-only responding for numeric values.
Numeric hallucination guardrail — post-processing step extracts all numbers from LLM response and verifies each exists verbatim in the source context passed in. If any fails verification, graceful fallback is triggered instead.
SSE streaming response — answer streams token-by-token from FastAPI to React via Server-Sent Events. First token must appear within 2 seconds of query submission.
Source attribution panel — collapsible panel beneath each answer showing 1–3 citation references (source name, type, short excerpt). Populated from citation JSON in API response.
Graceful failure handler — when guardrail triggers or classifier cannot route: system returns a confident, non-technical message stating what it cannot answer and inviting rephrasing. Zero Python errors visible to user. Zero hallucinated numbers ever rendered.
Suggested questions (5 chips) — pre-validated demo questions displayed as clickable chips. Each populates the input and triggers the live query. All 5 have pre-validated good answers in the test suite.
Demo Mode toggle — hidden toggle (keyboard shortcut or URL param) that serves pre-cached responses to the 5 scripted questions with client-side streaming simulation. Fallback for API latency or WiFi issues during pitch. Off-script queries always use live system.
Nice to have only if all 5 scripted Qs pass by Day 7
Proactive alert feed — 2–3 pre-generated insight cards surfaced above the chat input (e.g., "NZ fresh produce margin dropped 4% this week"). Powerful for showing push intelligence. Do not build until core is solid.
Role switcher — lightweight toggle between CEO view and Regional Manager view showing same data with different scope. Excellent for the pitch narrative. One afternoon if the core is done.
Follow-up question context — retain the previous answer in the LLM context so the CEO can ask a clarifying follow-up without re-stating the question. Adds depth to the conversation feel.
Out of scope explicitly excluded — do not build
Live API connections to SAP, Salesforce, or any real client system
User authentication, access control, or session management
Mobile interface or responsive layout below 1024px
Full dynamic NL-to-SQL engine — this is Phase 2 in the paid engagement
WhatsApp or Teams bot interface
Data persistence, user history, or conversation storage
Production error handling, logging, or monitoring
Multi-language support
03 Key user stories

Primary persona: James Whitfield, CEO, FreshChoice Group. Finance and operations background. Highly commercial. Sceptical of conceptual pitches. Will type his own questions. Must be convinced in 20 minutes.

US-01 Ask a structured data question must have
As the CEO, I want to type "How did fresh produce margins perform in New Zealand last month versus the prior quarter?" and receive an answer with actual percentage figures and a short explanation of what drove the change — without waiting for my finance team.
Acceptance criteria
Response includes at least two numeric values and a trend direction (up/down/flat)
A 1–2 sentence explanation of the driver is included alongside the numbers
First token appears within 2 seconds. Full response within 10 seconds.
Source attribution panel cites the structured data fixture by name
US-02 Ask about what teams are saying must have
As the CEO, I want to type "What are the top issues my regional managers are escalating this week?" and receive a synthesised summary drawn from Teams and email messages — not a raw list of messages, but an intelligent distillation of themes.
Acceptance criteria
Response summarises 2–3 distinct themes from the unstructured corpus
At least one citation references a specific source with date (e.g., "Auckland regional manager Teams thread, 14 March")
Response reads as synthesised analysis, not a message dump
No numeric values appear in this response unless they were present verbatim in the corpus
US-03 Ask a blended cross-source question must have — the wow moment
As the CEO, I want to ask "What's happening with supply chain disruptions and how is it affecting our margins?" and get a single answer that connects the financial impact from our data with the operational context from what my team has been saying — without me having to ask two separate questions.
Acceptance criteria
Answer explicitly references both a number (from lookup) and a qualitative source (from RAG) in the same response
The two are causally connected in the narrative — not listed separately
Example of passing response: "The 3.2% margin drop in NZ correlates with the Auckland DC delay flagged by Sarah Chen on 11 March"
Source panel shows citations from both source types
All numbers in the response verified by the guardrail post-processor
US-04 See where the answer came from must have — trust signal
As a CEO who has been burned by inaccurate dashboards before, I want to see exactly which data file or message thread each part of the answer was drawn from — so I can trust the numbers without having to verify them myself.
Acceptance criteria
A collapsible "Sources" panel appears beneath every answer
Each source shows: name, type (structured data / Teams / email), and a short excerpt
Panel is collapsed by default, expanded on click — does not clutter the main answer
1–3 sources per answer. Never zero on a successful response.
US-05 Use suggested questions must have — demo safety net
As a demo participant, I want to see pre-loaded question chips I can click to trigger reliable, impressive answers — so the demo flows naturally even without scripting the CEO's every move.
Acceptance criteria
5 question chips visible above the input field on load
Clicking a chip populates the input and triggers the query (does not auto-submit — CEO presses Enter, keeping him in control)
All 5 chips have pre-validated passing answers confirmed during red-team
In Demo Mode, these 5 chips return pre-cached answers instantly
US-06 Handle off-script questions gracefully must have — added v0.2 new
As the CEO typing a question the system wasn't specifically built for, I want to receive a response that is honest about what it knows and doesn't know — rather than a hallucinated answer or a technical error on screen.
Acceptance criteria
Zero Python tracebacks or API error messages ever visible to the user under any condition
Zero hallucinated numeric values — enforced by the guardrail post-processor
When guardrail triggers: system returns a calm, confident message (e.g., "I don't have enough data to give you a reliable number on that — can you try rephrasing or ask about a specific market?")
Tested against 15 off-script queries during red-team. Must return plausible, non-embarrassing responses on at least 10 of 15.
US-07 See the answer stream in real time must have — added v0.2 new
As the CEO, I want to see the answer begin appearing immediately after I press Enter — not wait 12 seconds staring at a spinner — so the system feels alive and intelligent rather than slow.
Acceptance criteria
First token appears within 2 seconds of query submission in normal network conditions
Answer streams token-by-token with no visible chunks or jumps
Source panel populates after the prose completes — does not interrupt reading flow
In Demo Mode, streaming is simulated client-side at the same visual cadence as live streaming
04 Data requirements

All data is synthetic and fictional. No real FreshChoice data is required or expected. Data must be internally consistent — if NZ margins drop in week 3 of the structured fixture, the Teams messages must reference that same event. The PM will provide a structured authoring template to the DS on Day 1 morning. The 5 demo questions are locked before data authoring begins.

Data authoring is the highest-risk task in the sprint. It must start Day 1 and complete by end of Day 4. Poor data produces weak demo answers that no amount of prompt engineering can recover.

Dataset Owner Specification
Structured fixture 1
Sales & margin data
DS
500–800 rows. Weekly sales volume and gross margin % by: product category (fresh produce, dry goods, beverages, dairy), market (AU / NZ / SG), store format (hypermarket / supermarket / convenience). 14–16 weeks of history. Must include a visible margin dip in NZ fresh produce weeks 10–12 to support demo questions. Delivered as CSV + pre-parsed into lookup JSON.
Format: week_ending, market, category, format, revenue_aud, gross_margin_pct, units_sold
Structured fixture 2
Supply chain log
DS
20–40 rows. Incident records covering distribution disruptions. Fields: date, affected_dc (Auckland / Melbourne / Singapore), affected_categories (comma-separated), estimated_margin_impact_pct, status (resolved / ongoing), notes. Must include an Auckland DC event in week 11 that causally links to the NZ margin dip in Fixture 1.
Format: incident_date, dc_name, categories_affected, margin_impact_pct, status, summary_note
Unstructured corpus 1
Teams message threads
DS
4–6 threads, 8–15 messages each. Covering: (1) NZ fresh produce supply issues referencing the Auckland DC delay, (2) cross-market performance concerns, (3) e-commerce channel growth, (4) staff rostering escalation. Written in the authentic voice of a regional ops manager — direct, slightly informal, with specific names and dates. Named characters: Sarah Chen (CSO), regional managers for Auckland, Melbourne, and Singapore.
Format: thread_name, date, sender_name, sender_role, message_body
Unstructured corpus 2
Email summaries
DS
6–10 emails. From regional managers to Sarah Chen (CSO) or James Whitfield (CEO) covering: weekly ops summaries, supplier delay notifications, margin concern escalations, and one e-commerce performance update. Each email should have a clear subject, date, sender, and 3–6 paragraph body. Must cross-reference the Auckland DC event from Fixture 2.
Format: email_date, sender_name, sender_role, recipient, subject, body
Pre-computed lookup JSON
DS
40–50 query variants derived from the 5 demo questions and natural rephrasings. Each entry maps a query pattern (keyword set) to a pre-computed answer payload: the relevant data rows, the computed metric, and a short context note. Enables deterministic, millisecond responses to structured data questions. Must cover all 5 demo questions plus at least 6 plausible variations of each.
Format: query_keywords[], market?, category?, metric_type → answer_payload{value, trend, context, source_ref}
Embedding index
DS
All unstructured documents (Teams threads + emails) chunked at 200–350 tokens with 40-token overlap. Embedded using OpenAI text-embedding-3-small or equivalent. Stored in ChromaDB (local). Each chunk metadata must include: source_name, source_type (teams/email), date, sender_name. Top-k retrieval set at k=4 for demo conditions.
Target: <500ms retrieval latency on demo hardware
05 API contract — agreed Day 2

The following schema is agreed between DS and ENG by end of Day 2. No changes after Day 6 without PM sign-off. The engineer builds all UI components against this schema from Day 3 onward.

POST /api/query — request
{ "query": string, // raw user input "demo_mode": boolean, // if true, return pre-cached response "session_id": string | null // reserved for future context — pass null now }
SSE stream — response events
// Event 1..N: token stream event: token data: { "token": string } // Final event: metadata event: done data: { "query_type": "structured" | "unstructured" | "blended" | "fallback", "citations": Citation[], "guardrail_pass": boolean // false = graceful failure was served } // Citation object Citation { "source_name": string, // e.g. "NZ Sales Summary Mar 2025" "source_type": "structured_data" | "teams" | "email", "excerpt": string, // max 120 chars "date": string | null }
06 Technical architecture
Data scientist stack DS owns
Data authoring — synthetic fixtures from PM template. CSV → pandas → pre-computed lookup JSON. Days 1–4.
RAG pipeline — chunk + embed Teams/email corpus. ChromaDB local vector store. text-embedding-3-small. k=4 retrieval. Days 4–6.
Query classifier — keyword-based routing function (~30 lines Python). Structured keywords → lookup. Unstructured keywords → RAG. Unknown → both. Day 5.
System prompt + LLM call — single prompt that receives lookup data + RAG chunks + query. Returns prose + citation JSON. GPT-4o or Claude Sonnet via API. Day 6.
Numeric guardrail post-processor — regex extract all numbers from LLM response. Verify each exists in source context string. If any fail: return graceful fallback payload instead. Day 6. 0.5 days.
FastAPI SSE endpoint — POST /api/query. Streams token events then sends done event with citation JSON. Day 7.
Engineer stack ENG owns
React / Next.js web app — chat interface with message history, text input, send button. Clean, professional. Days 1–3.
SSE stream renderer — consume token events from FastAPI, append to message in real time. No visible chunking. Days 2–4.
Suggested question chips — 5 chips above input. Click → populate input field. CEO presses Enter. Days 3–4.
Source attribution panel — collapsible panel. Renders citation array from done event. Name, type badge, excerpt. Days 4–5.
Graceful failure UI — detects guardrail_pass: false in done event. Renders fallback message component. No error stack traces ever. Day 5.
Demo Mode — URL param ?demo=true or keyboard shortcut activates. Serves pre-cached response payloads for 5 scripted questions with simulated streaming at natural cadence. Day 5. Half day.
Full integration + smoke test — wire all components to live DS endpoint. Smoke test all 5 scripted questions end-to-end. Days 7–9.
07 Assumptions
A-01
Simulated data is accepted by the client
James Whitfield understands this is a pitch prototype, not a production system. The PM will frame this expectation explicitly at the start of the demo: "We've built this on a representative data model — connecting to your live SAP and Teams is what the paid engagement delivers."
Why: Live integration would consume 8–10 engineer days on connectors alone, leaving nothing for the UI, streaming, or polish. The CEO cannot tell the difference between well-designed synthetic data and real data in a 20-minute demo.
A-02
Pre-computed lookup is architecturally honest for the prototype
The lookup table replaces dynamic NL-to-SQL for the prototype only. Full NL-to-SQL is explicitly the first deliverable in the paid discovery phase. The PM owns the narrative bridge in the pitch: "What you see here uses a structured retrieval layer — in the engagement we'd connect this directly to your SAP queries in real time."
Why: Saves 3 DS days. CEO output is identical. Deterministic and debuggable for the demo. A live NL-to-SQL engine under 14-day time pressure is a reliability risk we cannot afford.
A-03
LLM called via API — no fine-tuning or self-hosting
We use GPT-4o or Claude Sonnet via their respective APIs. Decision locked Day 1. API keys provisioned before standup. No model switching after Day 3 — system prompt tuning is model-specific and re-tuning costs 1–2 days.
Why: Fine-tuning is weeks of work. API access is hours. Demo-quality output from a well-engineered system prompt is indistinguishable from a fine-tuned model to a non-technical audience.
A-04
Demo runs on a controlled environment — a single laptop
Prototype runs locally on the engineer's demo laptop, not in cloud infrastructure. Internet access is required for LLM API calls. Demo Mode pre-caches all 5 scripted answers as the fallback if WiFi is unreliable. Engineer is physically on-site during the pitch, running the machine.
Why: Removes the need for production hardening, deployment pipelines, load testing, and cloud security review — none of which are achievable in 14 days.
A-05
5 demo questions are locked by 9am Day 1
The PM delivers the 5 scripted demo questions and the data authoring template to the DS before 9am on Day 1. Data authoring cannot begin without them. Every hour of delay on Day 1 morning is an hour of risk at the end of the sprint. Questions are derived from James Whitfield's exact words in the SDR briefing memo.
Why: The entire data fixture, lookup table, and system prompt are designed around making these 5 questions produce impressive answers. They must drive the data design, not follow it.
A-06
Second DS is a named, available contingency
A second data scientist is identified by name before end of Day 1. They are briefable and available to start Day 5 if the Day 4 checkpoint shows the RAG pipeline has not started. The Day 4 checkpoint condition: lookup JSON complete AND ChromaDB ingestion begun. If either is not true, second DS is activated without further discussion.
Why: The DS workstream has no buffer at 12.5 days across 14 calendar days. The contingency DS is insurance, not a plan B — the primary DS is expected to deliver. But the contingency must be real and named, not theoretical.
A-07
Demo Mode is not deceptive
Pre-caching the 5 scripted answers in Demo Mode is standard practice for enterprise software demos. The system genuinely does what it shows — Demo Mode simply removes API latency risk for the hero moments. Off-script questions always use the live system. The PM does not tell the CEO "this is real-time" for the cached responses — the framing is "ask me anything" while the suggested questions are the safety net.
Why: Every major enterprise software demo pre-caches its hero scenarios. The alternative — risking a 15-second blank screen or an API timeout in the room — is not acceptable.
08 Sprint gates & checkpoints

These are go/no-go decision points. If a gate fails, the escalation action is immediate — not deferred to the next standup.

End of Day 1
Foundation gate
5 demo questions locked. Data template handed to DS. LLM provider decided and API keys working. Second DS name confirmed. If any of these fail: PM escalates to Calibrax leadership same day.
End of Day 2
API contract gate
Response schema signed off by DS + ENG. TL chairs the session. Schema document shared to the squad Slack/Teams channel. No schema changes after Day 6 without PM approval.
End of Day 4
DS pipeline checkpoint
Lookup JSON complete. ChromaDB ingestion started. If either condition is not met: second DS activated immediately for Days 5–8. No grace period.
End of Day 7
Integration gate
At least 3 of 5 scripted questions returning passing answers end-to-end. If fewer than 3 pass: scope review meeting with TL + PM. Nice-to-haves are cancelled. If fewer than 2 pass: escalate to Calibrax leadership.
End of Day 12
Red-team gate — pitch go/no-go
All 5 scripted questions pass. Zero visible errors. Zero hallucinated numbers. At least 10 of 15 off-script queries return plausible responses. If gate fails: Day 13 is used for emergency fixes, not rehearsal. PM decides whether to pitch.
End of Day 13
Code freeze
No code changes after Day 13 without joint TL + PM approval. Demo environment locked. Demo Mode loaded and tested on pitch laptop. Engineer confirms environment runs cleanly on venue hardware.
